How to update ermeeth

Anissa Saumtally

1 Introduction

1.1 The ermeeth package

  • ermeeth is an R package created by the ThreeME team for the ThreeME team

  • It is meant to contain all side functions and tools used to run and analyze ThreeME.

  • It is not meant to be published freely on the CRAN (it would not really work anyways).

  • Anyone in the team can modify it, provided they know what they are doing (i.e. don’t break anything please)

  • Luckily this little guide will give you the asics to modify and add functions

1.2 A few steps

  1. Get ermeeth from GitHub (Section 2.1)

  2. Modify (Section 3.1) or add your function (Section 3.2)

  3. Document your function (Section 3.3)

  4. Check that all is clean (checklist at Section 3.4 )

  5. Package building preparation (Section 4)

  6. Commit to git and deploy (Section 5)

2 The Basics

2.1 Getting things ready

You will need to have the following:

  • Get the Repository from GitHub at this address: https://github.com/ThreeME-org/ermeeth

  • While you may do tests in a different branch, any updates you want to bring for the team must be done in the master branch.

  • The latest R version: https://cran.r-project.org

    • This is quite important as we want to avoind compatibility issues as R advances in versions

  • The corresponding Rtools installation if you are on Windows: https://cran.r-project.org/bin/windows/Rtools/

    • This means if the latest version of R is 4.2.3, you must have Rtools 4.2.x

  • The following R packages installed: devtools, roxygen2

install.packages(c("devtools","roxygen2"))

2.2 Understanding the ermeeth folder/repo/Rproject p.1

ermeeth is not a simple Rproject, it is a package Rproject. This means that some folders have specific usages, and its general architecture should not really be modified unless truly necessary.

  • NAMESPACE: contains some package info, is modified automatically by the use of a command. Do not touch this

  • DESCRIPTION: the only field to manually change for now is Version: as the package is modified

  • dev_history.R: Contains useful commands for package building, good to keep track of programming steps regarding the package. It is not a script used for testing stuff

  • .Rbuildignore: files and folders to be ignored when building the package, not really to be modified unless you brought consequential changes to the file system. (but why would you?)

  • todolist: a text file to keep track of things to do.

  • LICENSE.md: the user license markdown, generated automatically but modifiable if needed

2.3 Understanding the ermeeth folder/repo/Rproject p.2

  • data-raw: folder to keep scripts where datasets attached to the package are created Advanced users only
  • inst: folder that contains all files you want to attach to the package (csv, rds, xlsx, zip). Parsimony is key though
  • man: where the function documentation is compiled. It is generated automatically in markdown format, no need to modify here.
  • tests: where you can do anything you want to test functions before bringing them to the package. What happens in tests stays in tests. Feel free to add folders and such.
  • R: Where the magic happens .the packaged functions are stored there. The rule is one function, one script. (some exceptions apply). Those R scripts are written with some rules that we will see later.
  • vignettes: someday the vignette will return and will be stored there. For now the vignette is in tests and should still be updated.

3 Updating ermeeth functions

3.1 Modifying an existing function

  • Once you did the necessary tests in the tests folder, open the R script of the function you want to modify in the R folder.

  • Modify the body of the function, and if needed functions arguments definitions.

    • As you write the function, if you call on functions from external packages, write them with the package::function syntax. ie :
what_time_is_it <- function(){
  cat(stringr::str_c("Today is ", Sys.Date() ) )
}

  • Do not leave active script lines except that of the function

    • here we commented the usual library call as well as a test example we might have written while building the function.
# library(stringr) 

what_time_is_it <- function(name){
  cat(stringr::str_c("Today is ", Sys.Date(), ", ",name ) )
}

# testing the function:
# name <- "Meriem"
# what_time_is_it()

Then you are ready to proceed to the function documentation Section 3.3

3.2 Adding a function

If you are creating a new function, first build it in the tests folder, test it, and once it works :

  • create a new R script in the R folder with the name of the function in the file name to find it easily later on.

  • make sure your function is built cleanly, with best practices in mind (ie no call to external objects and other rules we may have mentioned at some point)

  • As you write the function, if you call on functions from external packages, write them with the package::function syntax.

  • copy only the function lines in that script and save.

    • If you want to keep testing lines, comment them.

    • i.e. your script should have only those lines active

# library(stringr) 

what_time_is_it <- function(name){
  cat(stringr::str_c("Today is ", Sys.Date(), ", ",name ) )
}

# testing the function:
# name <- "Meriem"
# what_time_is_it()

Then you are ready to proceed to the function documentation Section 3.3

3.3 Function documentation

Do not skip this step : Once the function is written and stored in the R folder, it must be properly documented.

What does documented mean?

  • The documentation precedes the function in the form of roxygen comments. The lines start with #' (as opposed to normal comments that only start #), then different fields start with @.

  • Fortunately, this is semi-automatically generated

    • start by placing the text cursor anywhere in the function lines (has to be between the function’s curly brackets)
    • use the shortcut : ctrl + alt + shift + R (or click in the bar menu code/insert Roxygen Skeleton)
    • Part of the documentation should be generated on top of the function
    • If the documentation existed already, this procedure will updates @param fields if the function arguments were changed

once generated, fields must be completed manually, and some must be added manually

3.3.1 Filling in the documentation

While some documentation parameters can be generated automatically, they must be explicited by the function writer. Some fields must be added manually.

Existing fields

  • Replace Title with a title for the function

  • in the empty line, a longer function description, you can add @description, at the beginning (optional but appreciated)

  • @params {argument} : function arguments. The good practice is to state the class of the argument (ie data.frame, boolean, vector etc), the what it does, and finally what is the default option as written in the function

  • @return Write what the function returns upon execution (class, content)

  • @export leave empty

  • @example Write a quick example that can be copied, pasted and executed on a empty environment. Wrap your example with \dontrun{} if the function does complicated things that take time and cannot be visualised in the console. See picture on the right for examples.

3.3.2 Fields to add : @import and @importFrom

Many fields can be added to the roxygen documentation. They are not described here. However two important ones must be added if used in the function.

If your function calls for function from non-base packages (packages not loaded automatically in R), you must use the syntax package::function in your code, and declare them in the skeleton via @import or @importFrom

  • @import : used to declare an entire package
    • Used if many functionalities of the package are used, but in practice, unless maybe for dplyr or ggplot2, it is not worth using because importing an entire packages loads all the functions, which can create conflicts

-@importFrom : used to declare select functions from a package. See usage below :

#' @import dplyr
#' @import ggplot2
#' @importFrom purrr imap set_names map_dbl

## in this case dplyr and ggplot2 are loaded entirely, no need to declare selected functions. 
# For the purrr package only the three stated functions will be loaded

3.4 Checklist

While modifying or adding a function, did you do the following:

  • Made sure the roxygen documentation is up-to-date with any new function arguments you may have added or removed
  • Declared the packages for every non-basic function used ? ie used the package::function syntax.
  • Declared the imports in the roxygen documentation
  • Created a reproducible example in the roxygen field (and used \dontrun{} where appropriate.)
  • Did not leave any active line of code in the function script file besides the function and its documentation.
  • Your function is clean and devoid of any calls to external objects.

3.5 Optional (but greatly appreciated): The vignette

While examples in the functions documentation should be succinct, it is nice to have a more complete documentation in the vignette. The goal is to demonstrate more elaborate usages of your function, with more significant examples.

  • To do this, open the Functions_presentation.Rmd file in the tests folder (temporary, this will change later)

  • Add a section with your function. Include text explanation as well as sample code.

  • Don’t use external data, use either sample databases like iris or an ermeeth database like oilprice_fra_c28_s32.rds.

    • the latter is already loaded in the .Rmd with the name data

  • Make sure the markdown compiles after adding your section via a knit : ctrl + shift + k

4 Package buidling steps

4.1 Prepare to compile

  • Open the DESCRIPTION file and change the version : x.y.zz.a
    • if you’re adding a function increment the zz number and set z to 0
    • if you’re modifying a function increment the a. If a >= 9, then increment zz and set a to 0.
    • The idea is that when sorted alphabetically, the new version is seen as superior alphabetically speaking.
  • If you added calls in the @import or @importfrom fields to packages that were previously not called upon, you have to declare them:
    • open the dev_history.R script
    • add a line below similar lines with the following command and execute it (ctrl + enter, save the modified file)
usethis::use_package("Name of package used")

4.2 Generate automated files

  • Clear your environment
  • Generate function documentation via roxygen2::roxygenise() and devtools::document() (I don’t know if the latter is necessary, as it is a wrapper for `roxygenise``).

or for the lazy, copy and execute this piece :

rm(list= ls())
roxygen2::roxygenise()
devtools::document()

4.3 Final and crucial step: The check button

Your ultimate test: Did you break everything? Let’s see:

  • Find the Build tab :

  • Click the Check button

  • Wait a bit… have a sip of coffee or water

4.4 Check Success or Failure?

Look at the check summary. Warnings, or worse, errors, are not good for packages. You want to have 0 errors and 0 warnings. Unfortunately 0 notes will be difficult to reach for ermeeth but that is ok, we can live with it.

Success means:

Congratulations! You didn’t break anything - Commit your changes

  • You may proceed to the deployment step (Section 5).

The face of failure:

  • Investigate the source of warning/errors and fix it, then re-run the check. Do so until you fix them.

  • Fortunately, R tells you where it detected an issue.

    • Scroll up on the check tab: find some colored line that ends with WARNING or ERROR

  • read through it and solve it.

5 Deploying the new version

5.1 Install the package

  • You should have now a sound package.

  • First step: install the package

    • In the Build tab, click on the install button and choose Clean and Install.
    • This will install on your computer the new version of ermeeth that you’ve just created.
    • For it to be effective on your other RStudio opened instances, restart them (Session/Restart R).

5.2 Build the package

  • Second step: Build the source package (tar.gz file)
    • In the Build tab, click on the More button and choose Build source package.
    • This will create an ermeeth_x.y.zz.a.tar.gz file in the parent folder of the ermeeth Rproject
    • This part of the deployment step as well as the next one will be changed soon for something more practical

5.3 Deploy via GitHub

Technically, we should be able to install via github, but due to some limitations at the OFCE (and maybe at other institutions too), we use a less practical way.

  • Copy and paste the newly created tar.gz file in the src folder in any ThreeME branch of the ThreeME V3 repository.

  • Commit and push this change. Make sure the only change in this commit is the addition of the new ermeeth version.

  • Cherry-pick the commit in the active ThreeME branches, so that people currently working on other branches can access your changes.
